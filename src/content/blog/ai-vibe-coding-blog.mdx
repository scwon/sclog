# 서론: 개발자가 코드를 안 짜도 되는 시대

2025년, 나는 블로그를 만들고 싶었다.

예전 같았으면 이랬을 거다:
1. Next.js vs Astro 고민하며 3일
2. 디자인 시스템 만들다가 일주일
3. 다크모드 구현하다 버그 잡느라 이틀
4. 결국 "나중에 해야지" 하고 접음

근데 이번엔 달랐다. **Claude Code**라는 녀석이 있었다.

```
나: "블로그 만들어줘"
Claude: "넵"
```

진짜 이게 끝이었다.

---

# 뭘 만들었나

이 블로그가 그 결과물이다. 구체적으로:

- **Astro 5 기반 정적 블로그** - SSG로 빠르고 SEO 좋음
- **라이트/다크 모드** - 눈 아픈 사람 배려
- **MDX 지원** - 마크다운에 컴포넌트 섞기
- **포트폴리오 홈** - 프로필, 경력, 소셜 링크
- **읽기 경험 개선** - 목차, 코드 복사, 진행률 바, 맨 위로 버튼
- **3D 프로필 호버** - 동전처럼 뒤집히는 애니메이션 (이건 좀 멋있음)

전부 AI가 짰다. 나는 옆에서 "오 좋은데?", "아 그거 말고", "3D로 해줘" 이런 말만 했다.

# Speckit: AI한테 일 시키는 방법

그냥 "블로그 만들어"라고 하면 AI가 뭘 만들지 모른다. 그래서 **Speckit**이라는 걸 썼다.

## 워크플로우

```
/speckit.specify 읽기 경험 개선 기능 추가해줘
```

이러면 Claude가 알아서:
1. **스펙 문서** 작성 (뭘 만들건지)
2. **기술 리서치** (어떻게 만들건지)
3. **태스크 분해** (뭐부터 할건지)
4. **구현** (진짜 코드 작성)

나는 중간중간 "ㅇㅋ", "고", "메인 머지행" 이런 말만 했다.

## 실제 대화 예시

**기능 1: 읽기 경험 개선**

```
나: "블로그 기능 확장하고 싶은데 아이디어 좀"
Claude: (아이디어 10개 제시)
나: "읽기 경험 개선 위주로"
Claude: (5개 필수 기능 정리)
나: "필수 기능들 추가해볼게"
Claude: (스펙 작성 → 계획 → 구현 → 완료)
```

**기능 2: 프로필 호버**

```
나: "프로필 사진에 마우스 올리면 좀 멋있는 애니메이션
     막 돌아가면서 다른 이미지로 바뀌면 좋겠어"
Claude: (구현)
나: "3D로 동전처럼 회전했으면 좋겠어!"
Claude: (CSS perspective + rotateY로 수정)
나: "메인 머지행"
```

이게 전부다. 프롬프트 엔지니어링? 몰라. 그냥 하고 싶은 말 했다.

# 결과물: 실제 생성된 코드

믿기 힘들겠지만, 이 코드들 전부 AI가 짰다.

## 3D 동전 뒤집기 CSS

```css
.profile-avatar {
  perspective: 1000px;
}

.avatar-default,
.avatar-alternate {
  backface-visibility: hidden;
  transition: transform 0.6s ease;
}

.avatar-alternate {
  transform: rotateY(180deg);
}

.profile-avatar:hover .avatar-default {
  transform: rotateY(180deg);
}

.profile-avatar:hover .avatar-alternate {
  transform: rotateY(360deg);
}
```

나: "3D로 해줘"
Claude: `perspective`, `backface-visibility`, `rotateY` 다 알아서 적용

## 접근성까지 챙김

```css
@media (prefers-reduced-motion: reduce) {
  .avatar-default,
  .avatar-alternate {
    transition: opacity 0.1s ease;
  }
  /* 회전 애니메이션 제거, 페이드만 */
}
```

나는 접근성 얘기 안 했는데 알아서 넣어줬다. Constitution(프로젝트 헌법)에 "접근성 중요"라고 써놨더니 자동으로 적용.

# 이게 가능한 이유: Context가 핵심

Claude Code가 잘 작동하는 이유는 **컨텍스트**다.

1. **CLAUDE.md** - 프로젝트 규칙 정의
2. **Constitution** - 코드 품질 원칙 (테스트, 린트, 접근성 등)
3. **Speckit 템플릿** - 일관된 문서화

AI가 코드베이스 전체를 이해하고 있으니까:
- 기존 컴포넌트 스타일 따라감
- 디자인 토큰 자동 적용
- 타입스크립트 strict mode 준수

"여기선 이렇게 해" 같은 말 안 해도 알아서 맞춤.

# 솔직한 후기

## 좋았던 점

- **속도**: 평소 일주일 걸릴 작업을 하루만에
- **품질**: 내가 짜는 것보다 일관성 있음
- **학습**: AI 코드 보면서 "아 이렇게도 하는구나" 배움

## 아쉬웠던 점

- 가끔 context 날아가서 처음부터 설명해야 함
- 복잡한 로직은 아직 사람이 검토 필요
- "이거 해줘"보다 "이거 하지 마"가 더 어려움

## 결론

**코드 짜는 시대는 끝났다**... 는 아직 아니고.

**코드 짜는 방식이 바뀌고 있다**가 맞는 것 같다.

나는 이제:
- 무엇을 만들지 **결정**
- 어떤 방향으로 갈지 **판단**
- 결과물이 괜찮은지 **검토**

이 세 가지에 집중하고, 타이핑은 AI가 한다.
